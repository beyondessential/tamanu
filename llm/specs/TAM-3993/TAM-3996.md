# TAM-3996 — Viewing change log

**Linear:** TAM-3996  
**Parent:** TAM-3993 (program form edit on desktop)  
**Shared requirements:** **@llm/specs/TAM-3993/index.md**  
**Related spec:** [TAM-3992](./TAM-3992.md) (edit flow, per-question “view change log”, list edited state)

## Summary

On **desktop web**, add **Change log** to the program form overflow menu, always load changelog data with **`GET /surveyResponse/:surveyResponseId/changes`**, and show **survey-response-level** edit history (newest first). Admin and “always on” rules: **@llm/specs/TAM-3993/index.md**.

---

## Shared requirements

See **@llm/specs/TAM-3993/index.md** for persistence, edit gating, changelog scope, **`GET /surveyResponse/:surveyResponseId/changes`**, `logs.changes` / sync, mobile, reporting, and changelog-always-on / admin setting removal.

---

## Overflow menu

- Add **Change log** to the forms overflow menu.
- **Order:** **Print, Edit, Delete, Change log** (same row as TAM-3992 **Edit**).
- **Change log** is **always visible** (not restricted to responses that have already been edited).
- If there is **no** edit history for that response (nothing to show from `logs.changes` for that scope), render **`<ContentUnavailableView>`** (or project equivalent), not an error state.

---

## API

- Endpoint and single-source rules: **@llm/specs/TAM-3993/index.md**.
- Changelog data MUST be fetched with:

  **`GET /surveyResponse/:surveyResponseId/changes`**

- Response shape is defined in implementation but MUST support:
  - **Response-level** history (single list for the whole submission).
  - **Newest-first** ordering for the user-visible list.
  - Enough detail to render **edited from** / **edited to** (or Figma-equivalent) per affected field within each logical change / revision.

- Register the route consistently with existing facility **`surveyResponse`** / `apiv1` patterns.

---

## Permissions

- **Read** permission on the form/program context is sufficient to open **Change log** and to use **view change log** from the response detail (TAM-3992).
- Read-only users can still **read** edit history when permitted to read the response.

---

## Changelog presentation

### Granularity

- **Survey response level:** one modal (or screen) per **`surveyResponseId`**, not a separate changelog per answer.

### Ordering

- **Newest edits first** (reverse chronological) for the combined history of that response.

### Value rendering (human-readable)

- Empty / null: **`<em>No answer</em>`** (or styled equivalent).
- **Multi-select:** **`<ul>`** of selected labels.
- **Dates / times:** **`<DateDisplay>`**, **`<TimeDisplay>`**, **`<DateTimeDisplay>`** as appropriate.
- **Files:** **filename** (and existing download behaviour if any).
- **Calculated fields:** show the **calculated value** as at that revision.

---

## Admin / configuration

Per **@llm/specs/TAM-3993/index.md** (always on; remove `audit.changes.enabled` / **`useSetting`** gating and **`is_audit_changes_enabled`**; remove admin toggle). **This issue owns** implementing admin UI removal and dead code paths for that gating.

---

## Acceptance criteria

1. **Change log** in overflow always shown; with no history, user sees **`ContentUnavailableView`**.
2. Client loads changelog with **`GET /surveyResponse/:surveyResponseId/changes`**.
3. User with **read** permission can open the changelog and see **from / to** style history where it exists.
4. Changelog is **response-level** only (same endpoint and content as TAM-3992 inline “view change log”).
5. **`audit.changes.enabled`** / **`is_audit_changes_enabled`** no longer controls this changelog; admin UI and dead paths removed or migrated per engineering standards.
6. No assumption that **`logs.changes`** exists on other facilities for synced copies of a response.
