# TAM-3992 — Editing a form & viewing edited form

**Linear:** TAM-3992  
**Parent:** TAM-3993 (program form edit on desktop)  
**Shared requirements:** **@llm/specs/TAM-3993/index.md**  
**Related spec:** [TAM-3996](./TAM-3996.md) (overflow change log, presentation rules, admin deliverables)

## Summary

On **desktop web**, allow users to **edit** an existing program form submission in place, enforce **dirty-only** resubmit, and surface **edited** state in lists and response views. **“View change log”** opens the **survey-response-level** changelog (same as overflow **Change log**; see TAM-3996). Shared persistence, API, sync, mobile, reporting, and admin rules are in **@llm/specs/TAM-3993/index.md**.

## Editing a form

### UI

- Add **Edit** to the program form / response overflow menu. **Menu order** (with TAM-3996): **Print, Edit, Delete, Change log**.
- **Edit** opens the standard survey completion UI with **previously persisted** answers visible and editable.
- User completes the **normal submission flow** to save edits.

### Behaviour

- **Unlimited** edits allowed.
- Submit control is **disabled** until the form is **dirty** (user cannot submit an unchanged survey response).
- **No** persistence on navigate-away or implicit save; only **explicit submit** writes changes.

### Permissions

- **Facility implementation:** program form edits use **`write` on the survey** (same check as `PATCH /surveyResponse/:id` on the facility server), not `create` on `SurveyResponse`. Overflow **Edit** is shown when the user has survey **write** for that row’s `surveyId`. (Product wording “same as submit” is interpreted in line with this facility permission model.)
- Not restricted to the original author; anyone with the above permission for that form context may edit.

### Data

- Updates **in place** on existing `survey_responses` and `survey_response_answers` records. Changelog sourcing: **@llm/specs/TAM-3993/index.md**; presentation: [TAM-3996](./TAM-3996.md).

## Viewing an edited form

### List / table

- An **edited** indicator displays next to the form name (and table shows an edited key) **per design**.
- **Open item:** Behaviour for when/how the table “edited” key appears should follow **MAR-style logic** where product requires parity; **exact MAR rules TBD** — track separately if needed.

### Response detail

- Where the standard UI lists answers, **per-question** copy may indicate that a given line was edited (granularity of indicator follows design / data from `logs.changes` on answers).
- **“Edited — view change log”** (or equivalent per Figma): opens the **single survey-response changelog** for that submission — **not** an answer-scoped log. Load data with **`GET /surveyResponse/:surveyResponseId/changes`**.
- **No** scroll-to-target, **no** highlight of a specific question or revision inside the changelog (unless explicitly added later).

### Changelog entry points vs TAM-3996

- Overflow **Change log** and inline **view change log** must show the **same** response-level changelog content, backed by the **same** endpoint.

## Acceptance criteria

1. User with appropriate **submit/edit** permission can open **Edit**, change at least one answer, and **submit**; DB updates the same `survey_responses` / `survey_response_answers` rows; **`end_time`** unchanged; **`updated_at`** advances as usual.
2. Submit remains **disabled** when the form matches persisted answers (**not dirty**).
3. Leaving the flow without submit does **not** persist edits.
4. List/detail UI matches **Figma** for edited affordances; table edited key **TBD vs MAR** if product still deferred.
5. **“View change log”** calls **`GET /surveyResponse/:surveyResponseId/changes`** and displays the **response-level** changelog (same as TAM-3996); no answer-only changelog route.
