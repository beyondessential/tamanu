# SOP: Disable FHIR job generation (dev-OTS)

Stop new FHIR jobs being generated by disabling the `fhir_refresh` triggers, then
empty the queue. This is a **developer** procedure: both building blocks (mass
`ALTER TABLE ... DISABLE TRIGGER fhir_refresh` and `TRUNCATE fhir.jobs`) are
**[dev-OTS]** — recoverable, but consequential and easy to leave half-done, so
they silently stop materialisation. Support does **not** run this alone — it is
escalate-only, requiring a **[dev-OTS]** and the specific context for why FHIR is
being turned off (see the Linear issue `TAMOC-267` referenced by the source).

Read `../ruled-out-actions.md` first. This SOP documents the steps so a developer
and support officer can run them together with eyes open; it is not a green light
for a first-line action.

## When this is appropriate

Only when FHIR materialisation is genuinely not in use on the deployment (or is
being deliberately turned off) and the job queue is filling with work nothing
consumes. If FHIR *is* in use, do not disable it — drain the queue instead
(`../runbooks/fhir-queue-backlog.md`).

## Steps (developer-run, with OTS)

1. Open psql in read/write mode: `bestool tamanu psql -W`
   (`../sops/connect-psql.md`).
2. Disable the `fhir_refresh` trigger on every upstream table that carries it.
   The set of tables is deployment/version-specific — a developer confirms the
   current list rather than pasting a stale one. It covers the clinical and
   reference tables that feed FHIR (for example `patients`, `encounters`,
   `lab_requests`, `imaging_requests`, `service_requests`' upstreams, and so on).
   Each statement is:

   ```sql
   ALTER TABLE <table> DISABLE TRIGGER fhir_refresh;
   ```

   `[unverified]` — the exact table list is not pinned here on purpose; it drifts
   between versions. A developer generates the current list before running.
3. Empty the queue:

   ```sql
   TRUNCATE fhir.jobs;
   commit;
   ```

## Reversing it

Re-enable every trigger you disabled (`ENABLE TRIGGER fhir_refresh`) — leaving
them disabled silently stops materialisation. Track exactly which tables were
touched so none is missed. Then let the worker rebuild the queue as normal.

To turn FHIR fully back on, also check the config/worker toggles
(`../sops/disable-materialised-resources.md` covers the per-resource settings and
`integrations.fhir.*` flags).

## How the materialiser catches up (does it need a full rebuild?)

Short answer: materialisation is **trigger-driven, not cursor-based**. There is
**no watermark** the worker resumes from. Confirmed in code:

- The `fhir_refresh` trigger fires `AFTER INSERT OR UPDATE OR DELETE` on each
  upstream table and enqueues a `fhir.refresh.fromUpstream` job for that row
  (`packages/database/src/services/setFhirRefreshTriggers.js`). The worker just
  drains the queue (`packages/shared/src/tasks/fhir/refresh/fromUpstream.js`) —
  the queue itself is the only record of "what's left", so truncating it loses
  that.
- Recovery for **missing** resources is a scheduled reconciliation, not a cursor.
  `FhirMissingResources` (`packages/shared/src/tasks/fhir/FhirMissingResources.js`,
  scheduled daily by default — `config.schedules.fhirMissingResources`,
  off-hours) LEFT JOINs each upstream table against its materialised
  `fhir.<resource>` table and enqueues **LOW**-priority refresh jobs for every
  upstream row that has **no** materialised row (`r.id IS NULL`).

So after re-enabling triggers and emptying the queue: rows that were **created but
never materialised** are picked up automatically by the next `FhirMissingResources`
run (once a day, or sooner if a developer triggers it) — no manual full rebuild
needed for those.

**Important limitation:** `FhirMissingResources` only finds *entirely missing*
resources. A row that was **updated** while the triggers were off — its
`fhir.<resource>` row still exists but is now **stale** — is invisible to the
reconciliation and will **not** be refreshed. Those need an explicit
re-materialisation: re-touch the upstream row so the trigger re-fires, or run a
targeted/full rebuild. `[pending — Rohan/dev to confirm]` the standard operational
way to force a full re-materialisation for the affected window; the mechanism
above is certain from the code, the "force a full rebuild" procedure is a dev
call.
